# Agent X Enterprise Re-architecture Checklist (2026)

> Generated by Master CTO. This document outlines the step-by-step refactor
> required to elevate the Agent X module (`backend/src/modules/agent`) from a
> static rules engine to a robust, scalable, enterprise-grade AI orchestrator.

## Phase 1: Core Foundation & Execution Strictness

### 1. The Context Builder (`memory/context-builder.ts`)

- [ ] **Rip out hardcoded dictionaries:** Remove static configurations like
      `SPORT_SEASONS` and `ROLE_PERSONAS`.
- [ ] **Dynamic Hydration:** Modify the Context Builder to fetch rules
      dynamically from a `PlatformConfig` or `AgentConfig` collection in
      Firestore, integrating `user.agentSettings.persona`.
- [ ] **Result:** A purely functional pipeline that pulls state at runtime
      without requiring core code deployments.

### 2. The LLM Execution Layer (`llm/openrouter.service.ts`)

- [ ] **Enforce Structured Outputs:** Integrate **Zod** to define exact
      TypeScript shapes constraints (e.g.,
      `z.object({ insights: z.array(...) })`).
- [ ] **Native JSON Schemas:** Pass these schemas directly to the OpenRouter API
      via `response_format: { type: "json_schema" }`.
- [ ] **Remove Hacks:** Eradicate fragile string-parsing logic entirely (e.g.,
      `stripMarkdownFences`).

### 3. The Orchestrator (`services/generation.service.ts`)

- [ ] **Decouple the Pipeline:** Ensure `generation.service.ts` purely handles
      the lifecycle:
  1. `context = await contextBuilder.buildDynamicContext(user)`
  2. `schema = PlaybookSchema`
  3. `llmService.generateWithSchema(prompt, schema)`
  4. `saveToFirestore(result)`

## Phase 2: Telemetry, Caching & Boundaries

### 4. State-Aware Semantic Cache (`memory/semantic-cache.service.ts`)

- [ ] **Enrich Semantic Cache Keys:** Expand the cache hash from just "prompt
      vectors".
- [ ] **State Parameters:** Include `userContextState`, `role permissions`, and
      `specific DB entity versions` in the cache key.
- [ ] **Proactive Invalidation:** If underlying data updates (e.g., stats
      change, compliance window shifts), invalidate matching semantic clusters.

### 5. Strict Tool & Skill Boundaries (`tools/` & `skills/`)

- [ ] **Remove Direct DB Access:** Audit and remove all direct raw
      Firestore/MongoDB queries inside Tools.
- [ ] **Thin Mapping Layers:** Tools must act exclusively as schema-to-service
      mappers. Rely on the LLM to map params, then invoke standard,
      strongly-typed internal Enterprise `Services` that naturally enforce AuthZ
      and aggregation.

### 6. DAG Pipeline Telemetry (`agent.router.ts`)

- [ ] **Embed the Four Pillars:** Inject Observability deeply into the Directed
      Acyclic Graph (DAG) task iterator.
- [ ] **Distributed Tracing:** Wrap every sub-agent step in
      `PerformanceService.trace(TRACE_NAMES.AGENT_TASK_EXECUTION)`. Emit
      localized segmented logging/breadcrumbs for surgical debugging.

## Phase 3: High Availability & Scaling

### 7. Intelligent Fallback Execution Matrix (`llm/openrouter.service.ts`)

- [ ] **Tiered Failovers:** Build an automatic model fallback mechanism
      constrained by Zod schema rules (e.g., `Claude 3.5 Sonnet` -> `GPT-4o` ->
      `Llama 3`).
- [ ] **Resilience:** Abstract retry states internally with exponential backoff
      to shield upper orchestrators from ephemeral LLM provider outages.

### 8. Queue Backpressure & Circuit Breakers (`queue.service.ts` & `pubsub.service.ts`)

- [ ] **Implement Backpressure:** Use Token-Bucket algorithms. If OpenRouter
      returns 429s or high latency spikes post-Friday night games, dynamically
      throttle Pub/Sub ACK rates.
- [ ] **Dead Letter Queues (DLQ):** Poison-pill queries (e.g., prompt injections
      causing parsing loops) must be fast-failed into a DLQ for manual
      inspection rather than gridlocking the worker nodes.
